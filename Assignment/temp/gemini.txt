If you look at the Print() method in your generated output.java, you'll see this:

    _t50 = j < size;    // <-- Condition evaluated ONCE before the loop

In your CodeGenVisitor.java, when you evaluate the WhileStatement, you extract the condition into a temporary variable (_t50) before the loop starts. However, you never update this temporary variable inside the loop body. Because _t50 stays true forever, j keeps incrementing indefinitely until number[j] exceeds the array's size of 20.
The Fix: Re-evaluate the Condition Inside the Loop

To fix this, you need to append the condition re-evaluation to the very end of your while loop block so that it updates your temporary variable on every iteration.

Here is the updated visit(WhileStatement n, CodeGenContext ctx) method for your CodeGenVisitor.java:

    /**
     * WhileStatement: "while" "(" Expression() ")" Statement()
     * f2=condition, f4=body
     * TacoJava: condition must be Identifier.
     */
    @Override
    public String visit(WhileStatement n, CodeGenContext ctx) {
        // 1. Evaluate condition before the loop
        String cond = normalizeExprToIdent(n.f2, ctx);
        
        ctx.emitStmt("while (" + cond + ") {");
        ctx.pushIndent();
        
        // 2. Execute the body of the loop
        n.f4.accept(this, ctx);
        
        // 3. Re-evaluate the expression at the end of the loop body
        // This emits any required setup temps into the loop body and 
        // assigns the fresh evaluation back to our initial condition variable.
        String nextCond = visitExpression(n.f2, ctx);
        ctx.emitStmt(cond + " = " + nextCond + ";");
        
        ctx.popIndent();
        ctx.emitStmt("}");
        return null;
    }

The Root Cause

When your visit(MethodDeclaration n, CodeGenContext ctx) evaluates the return expression, it tries to normalize it using toIdent().

toIdent calls isSimpleIdent("true"). Because "true" is composed entirely of letters, isSimpleIdent returns true. It thinks "true" is a valid variable name! Since it assumes it's already an identifier, it skips creating a temporary variable and just returns the literal string.
The Fix

Update the isSimpleIdent method at the very bottom of your CodeGenVisitor.java file to explicitly reject these keywords:

    /**
     * Returns true if s is a valid Java identifier (letter/$/_ followed by
     * letter/digit/$/_ only). Used to decide whether a temp is needed.
     */
    private boolean isSimpleIdent(String s) {
        if (s == null || s.isEmpty())
            return false;
            
        // ADD THIS CHECK: Reject literals and keywords
        if (s.equals("true") || s.equals("false") || s.equals("this") || s.equals("null")) {
            return false;
        }

        char first = s.charAt(0);
        if (!Character.isLetter(first) && first != '_' && first != '$')
            return false;

    }

    /**
     * Infer the return type of a MessageSend by looking up the method
     * in the symbol table using the inferred type of the RECEIVER.
     */
    private String inferMessageSendType(MessageSend ms, CodeGenContext ctx) {
        String methodName = ms.f2.f0.tokenImage;
        
        // 1. Infer the type of the receiver (e.g. 'n' in 'n.GetRight()')
        String receiverType = inferPrimType(ms.f0, ctx);
        
        // 2. Look up the method in the receiver's class hierarchy
        if (receiverType != null && !"Object".equals(receiverType)) {
            String ret = ctx.symTable.getMethodReturnType(receiverType, methodName);
            if (!"Object".equals(ret)) {
                return ret;
            }
        }
        
        // 3. Fallback: Search all classes (best-effort for untyped receivers)
        for (SymbolTable.ClassInfo ci : ctx.symTable.getAllClasses().values()) {
            SymbolTable.MethodInfo mi = ci.getMethod(methodName);
            if (mi != null)
                return mi.returnType;
        }
        return "Object";
    }

By calling inferPrimType(ms.f0, ctx), you correctly extract the type of the receiver. If the BuritoJava code says n.GetRight(), the compiler will realize n is of type "Tree".
